**작성일**: 2026-03-18

## 📌 개요

**SkillsBench**는 2026년 2월 발표된 **AI 에이전트의 "Skills"(스킬)이 실제로 성능을 향상시키는지 측정하는 첫 번째 벤치마크**입니다. 스킬은 LLM 에이전트에 도메인 특화 절차적 지식을 제공하는 구조화된 패키지로, Claude Code, Gemini CLI, Codex CLI 등 최신 AI 에이전트 도구에서 빠르게 채택되고 있습니다.

---

## 🎯 벤치마크 규모

- **84개 검증된 테스크** (원래 322개 제안 중 엄격한 선별)
- **11개 도메인** 커버:
  - Software Engineering
  - Healthcare (의료)
  - Finance (금융)
  - Cybersecurity (사이버보안)
  - Science (과학)
  - Manufacturing (제조)
  - Data Analysis, Mathematics, Energy, Economics, etc.
- **7,308개 실행 궤적** (trajectories) 기록
- **105명의 기여자** (학계 + 산업 참여)

#### 평가 조건 3가지
1. **No Skills**: 기본 지시사항만 제공
2. **Curated Skills** (수동 큐레이션): 도메인 전문가가 작성한 절차적 가이드
3. **Self-Generated Skills**: 모델 스스로 생성한 절차적 지식

## 📊 주요 벤치마크 결과

### Finding 1: 큐레이션된 스킬은 평균 +16.2pp 개선

| 모델/에이전트 | 개선도 |
|-----------|-------|
| Claude Code + Opus 4.5 | **+23.3pp** (최대) |
| Codex CLI + GPT-5.2 | +14.8pp |
| Gemini CLI + Gemini 3 Flash | +13.6pp |
| **평균** | **+16.2pp** |

**주목**: 높은 편차 존재 (+13.6pp ~ +23.3pp)

### Finding 2: 자체 생성 스킬은 효과 없음 (-1.3pp)

- 모델 스스로 생성한 스킬: **거의 도움이 안 됨**
  - Opus 4.6만 미약한 개선 (+1.4pp)
  - GPT-5.2: **-5.6pp 악화**
  - 나머지: 거의 변화 없음

**결론**: 효과적인 스킬은 인간 전문가의 도메인 지식이 필수

### Finding 3: 도메인별 편차 극심

| 도메인 | 성능 향상 | 특징 |
|-------|----------|------|
| **Healthcare** | **+51.9pp** 🏆 | 전문화된 절차 필요 |
| **Manufacturing** | **+41.9pp** | 도메인 특화 워크플로우 |
| **Finance** | +35.2pp | 금융 규정/절차 |
| **Cybersecurity** | +28.1pp | 보안 프로토콜 |
| **Software Engineering** | **+4.5pp** | 모델의 기본 지식 충분 |
| **Mathematics** | **+6.0pp** | 예측 가능한 패턴 |

💡 **해석**: 모델 사전학습에 덜 포함된 전문 도메인일수록 스킬 효과가 큼

### Finding 4: 개별 테스크 분석

**최대 개선 테스크**:
- Mario Coin Counting: +85.7pp (2.9% → 88.6%)
- Sales Pivot Analysis: +85.7pp
- Flood Risk Analysis: +77.1pp

**음수 효과 테스크** (16개/84개):
- Taxonomy Tree Merge: -39.3pp
- Energy AC Optimal Power Flow: -14.3pp
- 원인: 스킬의 충돌하는 가이드 또는 불필요한 복잡성

## 🔧 스킬 설계 원칙 (Design Insights)

### Finding 5: 2-3개 스킬이 최적

```
스킬 수별 성능:
- 2-3개: +18.6pp ✅ (최적)
- 4+개: +5.9pp (감소수익)
- 1개: +14.2pp
```

👉 **"적을수록 좋다"** - 과도한 스킬은 인지적 오버헤드 발생

### Finding 6: 간결함이 포괄성을 이김

```
스킬 길이별 성능:
- 상세한 (Detailed): +18.8pp ✅
- 컴팩트 (Compact): +17.1pp ✅
- 포괄적 (Comprehensive): -2.9pp ❌
```

💡 **교훈**: 단계별 절차 + 예제 1-2개 > 긴 설명서

### Finding 7: 모델 규모 보정

- **Claude Haiku 4.5 + 스킬** (27.7%)이
- **Claude Opus 4.5 무스킬** (22.0%)을 초과

→ **스킬이 모델 용량 부족을 부분적으로 보정 가능**

## 📈 벤치마크 방법론

### 엄격한 검증 프로세스

**자동 검사:**
- 구조 유효성 (파일 존재, TOML/YAML)
- 오라클 실행 (100% 성공 필수)
- AI 생성 콘텐츠 탐지 (GPTZero)
- 누수 감지 (스킬이 답을 직접 제시하면 거부)

**인간 검토** (도메인 전문가):
- 데이터 타당성 (합성 데이터 거부)
- 작업 현실성
- 스킬 품질
- 부정행위 방지

### 결정론적 검증
- pytest 기반 프로그래매틱 검증 (LLM-as-Judge 거부)
- 수치 허용도 포함
- 완전한 궤적 로깅

## 💡 실무 시사점

### 스킬 작성자를 위한 조언

✅ **하라:**
- 단계별 절차 명시 (2-3 스킬 모듈 최적)
- 작동하는 예제 1-2개 포함
- 포맷 제약 명시 (JSON 반복, 특정 명령어)

❌ **하지 말 것:**
- 긴 설명서 (문맥 낭비)
- 과도한 스킬 (4+개는 성능 저하)
- 일반적 팁 ("pandas 사용해" 수준)

### 에이전트 배포자를 위한 조언

1. **도메인 선택**: Healthcare, Manufacturing은 큰 개선 가능
2. **하네스 선택**: Claude Code가 스킬 활용 최적
3. **모델 트레이드오프**: 작은 모델 + 스킬 = 큰 모델 무스킬
4. **다중 평가**: 에이전트 간 스킬 효과 편차 고려

## 🎓 결론

**SkillsBench는**:

1. **첫 번째 대규모 스킬 평가 프레임워크** - 7,308개 궤적으로 입증

**2026년 AI 에이전트 시장에서 스킬은 선택이 아닌 필수 - 하지만 어떻게 설계하느냐가 성패를 결정**

**해설**: SkillsBench는 2026년 2월 발표된 첫 번째 스킬 평가 벤치마크예요. 스킬은 LLM 에이전트에 도메인 특화 절차적 지식을 제공하는 구조화된 패키지로, 단순히 존재하는 것이 아니라 정말로 성능을 끌어올리는지를 엄격하게 검증하는 것이 핵심 목표입니다.

**Q2. 사람이 직접 큐레이션한 스킬과 모델이 스스로 생성한 스킬, 둘 중 어느 쪽이 성능을 높였나요?**

<details markdown="1">
<summary>✅ 정답 보기</summary>

**정답**: 사람이 큐레이션한 스킬 (평균 +16.2pp 개선), 자체 생성 스킬은 효과 없음(-1.3pp)

**해설**: 도메인 전문가가 작성한 큐레이션 스킬은 평균 +16.2pp 개선을 보였지만, 모델이 스스로 만든 스킬은 거의 도움이 안 됐어요(평균 -1.3pp). GPT-5.2는 오히려 -5.6pp 악화됐죠. 결국 효과적인 스킬에는 인간 전문가의 도메인 지식이 필수라는 결론으로 이어집니다.

</details>

**Q3. (OX) 스킬은 소프트웨어 엔지니어링 같은 익숙한 도메인일수록 효과가 크다.**

<details markdown="1">
<summary>✅ 정답 보기</summary>

**정답**: X (틀림)

**해설**: 반대예요. Healthcare(+51.9pp), Manufacturing(+41.9pp)처럼 모델 사전학습에 덜 포함된 전문 도메인일수록 스킬 효과가 컸습니다. 반면 Software Engineering(+4.5pp), Mathematics(+6.0pp)처럼 모델이 이미 기본 지식을 충분히 갖춘 영역에서는 스킬의 개선폭이 작았어요.

</details>

**Q4. 스킬을 무작정 많이 넣으면 성능이 계속 올라갈까요?**

<details markdown="1">
<summary>✅ 정답 보기</summary>

**정답**: 아니요. 2-3개가 최적(+18.6pp)이고, 4개 이상은 오히려 감소(+5.9pp)합니다

**해설**: "적을수록 좋다"가 핵심 교훈이에요. 스킬 수가 2-3개일 때 가장 좋았고, 4개 이상부터는 감소 수익이 나타났습니다. 과도한 스킬은 인지적 오버헤드를 일으키기 때문이죠. 길이도 마찬가지여서, 상세하거나 컴팩트한 스킬은 효과적이지만 너무 포괄적인(Comprehensive) 스킬은 -2.9pp로 오히려 해가 됐어요.

</details>

**Q5. "작은 모델 + 스킬"이 "큰 모델 + 무스킬"을 이긴 사례가 있었나요?**

<details markdown="1">
<summary>✅ 정답 보기</summary>

**정답**: 네. Claude Haiku 4.5 + 스킬(27.7%)이 Claude Opus 4.5 무스킬(22.0%)을 초과했습니다

**해설**: 이 결과는 스킬이 모델 용량 부족을 부분적으로 보정할 수 있다는 걸 보여줘요. 즉 더 비싼 큰 모델을 쓰는 대신, 작은 모델에 잘 만든 스킬을 붙여서 비슷하거나 더 나은 성능을 낼 수 있다는 실무적 함의가 있습니다.

</details>

**Q6. 같은 모델이라도 어떤 에이전트 플랫폼(하네스)에서 돌리느냐가 스킬 효과에 영향을 줄까요?**

<details markdown="1">
<summary>✅ 정답 보기</summary>

**정답**: 네. 플랫폼마다 스킬 활용도가 달라 효과가 크게 갈립니다

**해설**: Claude Code는 네이티브 스킬 지원으로 활용도가 가장 높았고(Opus 4.5와 +23.3pp), Codex CLI는 스킬을 인정하면서도 실제로는 잘 활용하지 않는 경향이 있었어요. 그래서 SkillsBench는 모델뿐 아니라 에이전트 구현(하네스)도 성패에 중요하다는 점을 강조합니다.

</details>

**Q7. SkillsBench는 작업 성공 여부를 어떻게 판정했나요? LLM이 채점했을까요?**

<details markdown="1">
<summary>✅ 정답 보기</summary>

**정답**: LLM-as-Judge를 거부하고, pytest 기반 프로그래매틱(결정론적) 검증을 사용했습니다

**해설**: 채점을 LLM에 맡기면 일관성이 떨어질 수 있어서, SkillsBench는 수치 허용도를 포함한 pytest 기반 결정론적 검증으로 정답을 판정했어요. 여기에 오라클 실행 100% 성공 요구, GPTZero를 통한 AI 생성 콘텐츠 탐지, 스킬이 답을 직접 흘리는 누수 감지 등 엄격한 검증 프로세스를 거쳤습니다.

</details>

**Q8. (응용) 의료 분야 AI 에이전트를 배포하려는 팀에게 이 연구가 주는 조언을 한 가지 든다면?**

<details markdown="1">
<summary>✅ 정답 보기</summary>

**정답**: 전문가가 큐레이션한 2-3개의 간결한 스킬을 붙이면 큰 개선을 기대할 수 있다 (자체 생성 스킬에 의존하지 말 것)

**해설**: Healthcare는 스킬 효과가 +51.9pp로 가장 컸던 도메인이에요. 다만 모델이 스스로 만든 스킬은 효과가 없으므로 도메인 전문가가 직접 작성해야 하고, 너무 많거나 길지 않게 2-3개의 간결한 단계별 절차로 구성하는 것이 좋습니다. 하네스는 스킬 활용도가 높은 Claude Code 같은 플랫폼을 고르면 더 유리하고요.

</details>
