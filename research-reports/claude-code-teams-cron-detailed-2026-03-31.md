## 1. Agent Teams (멀티 에이전트 협업)

### 1.1 정의 및 작동 원리

**Agent Teams란:**
- 하나의 **팀 리더(Lead)** 세션이 여러 **팀원(Teammates)** 세션들을 조정하며 함께 작업하는 프레임워크
- 각 팀원은 **독립적인 Claude Code 인스턴스**로, 자체 컨텍스트 윈도우를 보유
- 팀원들은 **공유 작업 목록(Task List)** 과 **메시지 시스템(Mailbox)** 을 통해 협력

**핵심 아키텍처:**

```
┌─────────────────────────────────────────────────────┐
│                   Team Lead (Main Session)          │
│  - 팀 생성 및 조정                                  │
│  - 작업 할당 & 의존성 관리                          │
│  - 팀원 메시지 수신 & 결과 종합                     │
└────────┬────────────────────────────────┬───────────┘
         │                                │
    ┌────▼─────────┐          ┌──────────▼────┐
    │  Teammate 1  │          │  Teammate 2   │
    │  Context 1   │          │  Context 2    │
    │  Worker A    │          │  Worker B     │
    └────┬─────────┘          └────────┬──────┘
         │                             │
         └──────┬──────────────────────┘
                │
        ┌───────▼─────────┐
        │  Shared Task    │
        │  List & Mailbox │
        └─────────────────┘
```

**작동 원리:**

1. **팀 생성**: 사용자가 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` 활성화 후 요청
   ```
   "병렬 코드 리뷰를 위해 에이전트 팀 생성: 보안, 성능, 테스트 커버리지 리뷰어"
   ```

2. **팀원 스폰**: 리더가 작업 분석 → 필요한 팀원 수 결정 → 각 팀원 생성
   - 각 팀원은 **독립적인 컨텍스트 윈도우** 받음
   - 프로젝트 정보(`CLAUDE.md`, MCP 서버, 스킬) 자동 로드
   - **리더의 대화 이력은 전달되지 않음**

3. **작업 조정**: 공유 작업 목록을 통한 협업
   - 리더가 작업 생성 → 팀원들이 자동 청구 또는 리더가 할당
   - 작업 상태: Pending → In Progress → Completed
   - 의존성 자동 관리 (A 완료 → B 해제)

4. **직접 소통**: 각 팀원과 독립적 상호작용
   - `Shift+Down`: 팀원 순환
   - 메시지 송수신: 자동 전달, 리더를 경유하지 않음
   - 팀원들 간 직접 대화 가능

5. **종합 및 정리**: 리더가 결과 종합 → 팀 정리

```text
보안, 성능, 테스트 커버리지 리뷰어 3명을 팀원으로 구성해줄래.
```

**세부 지정:**
```text
다음 요구사항으로 팀을 만들어줄래:
- 팀 규모: 4명의 팀원
- 역할: 풀스택 리팩토링 (각 모듈 담당)
```


| **Reviewer** | 코드/설계 검토 | 독립적 평가 |

| **Investigator** | 문제 해결 | 가설 기반 탐사 |
| **Devil's Advocate** | 대안 제시 | 약점 지적 |

### 1.3 병렬 작업 관리

#### 작업 상태 및 흐름

```
Pending ──[청구]──> In Progress ──[완료]──> Completed
   ▲                   │                         │
   │                   │                         │
   └───────[거절]──────┘                         │
                                                 │
                    ┌─────────────────────────────┘
                    │
                    ▼
            [의존성 해제]
            (Blocked → Pending)
```

#### 작업 할당 전략

**방법 1: 리더가 명시적 할당**
```text
보안 리뷰어에게 인증 모듈 감사 작업을 할당해줄래.
성능 분석가에게는 DB 쿼리 최적화를 맡겨줘.
테스트 커버리지 체커에게는 엣지 케이스 검증을 할당해.
```

**방법 2: 자동 청구 (Self-claim)**
```text
각 팀원들이 완료 후 자동으로 다음 작업을 청구하도록 설정해줄래.
```

**의존성 관리:**
```text
프론트엔드 작업은 백엔드 완료 후 시작하도록 의존성 설정해줄래.
```

#### 작업 크기 및 분할

**최적 작업 크기:**
- **너무 작음**: 조정 오버헤드 > 병렬 이득
- **너무 큼**: 팀원 유휴 시간 증가, 수정 비용 증가
- **적절함**: **1팀원당 5-6개 작업** → 컨텍스트 스위칭 최소화

**작업 분할 예시:**

```
전체 프로젝트 (100시간)
│
├─ 보안 리뷰 (20시간)
│  ├─ 토큰 처리 감사 (5h) → Completed
│  ├─ 세션 관리 감사 (5h) → Completed
│  ├─ 입력 검증 확인 (10h) → In Progress
│  └─ 암호화 검토 (5h) → Pending (보안 승인 대기)
│
├─ 성능 프로파일링 (20시간)
│  ├─ DB 쿼리 최적화 (10h) → Completed
│  ├─ API 응답 시간 분석 (5h) → In Progress
│  └─ 캐싱 전략 개선 (5h) → Pending (쿼리 최적화 대기)
│
└─ 테스트 커버리지 (20시간)
   ├─ 단위 테스트 추가 (10h) → In Progress
   ├─ 통합 테스트 개선 (5h) → Pending (단위 테스트 대기)
   └─ E2E 시나리오 작성 (5h) → Pending
```

### 1.4 에이전트 간 통신

#### 메시징 시스템

**Message (1:1 통신):**
```text
[팀원 A에게만 전달]
보안 리뷰어에게 메시지: "토큰 처리 부분에서 refresh token rotation 확인해줄래?"
```

**Broadcast (1:All 통신):**
```text
[모든 팀원에게 전달 - 비용 주의]
모든 팀원에게: "인증 스키마가 변경되었어. 각자 영향받는 부분 다시 확인해줄래?"
```

#### 자동 메시지 전달

```
┌─────────────┐
│ Teammate A  │ ──────┐
│ (메시지 송신)│       │
└─────────────┘       │
                      ▼
                ┌──────────┐
                │ Mailbox  │ ──────> [Teammate B로 자동 전달]
                └──────────┘
                      ▲
                      │
┌─────────────┐       │
│ Teammate B  │───────┘
│ (메시지 수신)│
└─────────────┘
```

#### 통신 패턴 및 비용

| 패턴 | 비용 | 용도 | 주의사항 |
|------|------|------|---------|
| **Direct (1:1)** | 낮음 | 구체적 질문 | 리더 거치지 않음 |
| **Broadcast** | 높음 | 전체 동기화 | 팀원 수 배수 비용 |
| **Task List** | 낮음 | 작업 조정 | 자동 의존성 해제 |
| **Lead 대화** | 낮음 | 팀원 리다이렉트 | 리더가 중개 |

### 1.5 작업 조율 및 의존성 관리

#### 의존성 선언

**순차 작업:**
```text
다음 순서로 진행해줄래:
1. 데이터베이스 마이그레이션 (완료 필수)
   └─ 2. API 엔드포인트 업데이트 (1번 완료 후)
       └─ 3. 프론트엔드 업데이트 (2번 완료 후)
```

**병렬 작업 (의존성 없음):**
```text
이 3가지는 독립적으로 병렬 진행해줄래:
- 유닛 테스트 작성
- 문서 업데이트
- 성능 최적화
```

#### 의존성 자동 관리

```javascript
// 작업 상태 추적
{
  "task_1": {
    "status": "completed",
    "dependents": ["task_2", "task_3"]
  },
  "task_2": {
    "status": "pending",
    "depends_on": ["task_1"],
    "blocked": false  // task_1 완료 → 자동으로 false
  },
  "task_3": {
    "status": "pending",
    "depends_on": ["task_1", "task_2"],
    "blocked": true   // task_1은 완료, task_2는 진행 중 → 여전히 blocked
  }
}
```

#### 큰 의존성 체인 관리

**문제: 직선적 의존성은 병렬화 불가**
```
Task A → Task B → Task C → Task D → Task E
┌──────────────────────────────────────────┐
│ 총 시간: 모두 순차 (병렬화 이득 없음)      │
└──────────────────────────────────────────┘
```

**해결: 병렬화 가능한 작업 분해**
```
Task A (공통)
  ├─ Task B1 (경로 1) ┐
  │   └─ Task B1-i   ├─ Task E (병렬화 가능)
  ├─ Task B2 (경로 2) │
  │   └─ Task B2-ii  ┘
  └─ Task B3 (경로 3)
```

#### Hook을 이용한 품질 게이트

```text
팀원들이 작업 완료 후 리뷰 게이트를 설정해줄래.
```

**Hook 타입:**

| Hook | 트리거 | 활용 |
|------|--------|------|
| `TeammateIdle` | 팀원 유휴 시 | 최종 검증 |
| `TaskCreated` | 작업 생성 시 | 명세 검증 |
| `TaskCompleted` | 작업 완료 시 | 품질 검증 (Exit code 2로 거절) |

```bash
#!/bin/bash
# hooks/TaskCompleted.sh
if [[ "$TASK_TYPE" == "security-review" ]]; then
  if ! grep -q "CVE" "$REPORT_FILE"; then
    echo "보안 리뷰에는 CVE 분석이 필수입니다"
    exit 2  # 거절 → 팀원이 계속 작업
  fi
fi
```

### 1.6 실제 사용 사례

#### 사례 1: 병렬 코드 리뷰 (PR 감시)

**상황:** 100줄 PR을 3가지 관점에서 동시 검토

**팀 구성:**
```text
Create an agent team to review PR #142 in parallel.
Spawn three reviewers with distinct expertise:

1. Security Reviewer
   - 토큰 처리, 세션 관리 검증
   - 입력 검증 확인
   - CVE 위험도 평가

2. Performance Analyst
   - DB 쿼리 성능 분석
   - 메모리 사용 패턴
   - 병목 지점 도출

3. Test Coverage Checker
   - 새로운 엣지 케이스 발견
   - 테스트 누락 지점
   - 통합 테스트 필요성 판단

각자 독립적으로 리뷰 후 발견사항을 공유하고
충돌하는 의견은 토론으로 해결해줄래.
```

**결과:**
- **시간:** 3명이 병렬 → 50% 시간 단축
- **품질:** 3가지 렌즈로 100% 커버리지
- **비용:** 기존 3배 토큰 (시간 이득 > 토큰 비용)

#### 사례 2: 경쟁 가설 조사 (버그 분석)

**상황:** "앱이 한 메시지 후 연결 끊김" 버그 분석

**팀 구성:**
```text
Create an agent team to investigate the disconnect bug.
Spawn 5 investigators with competing hypotheses:

1. Network Layer Investigator
   - TCP 연결 상태 확인
   - 소켓 타임아웃 설정
   - 패킷 손실 패턴 분석

2. Protocol Handler Investigator
   - WebSocket keep-alive 검증
   - 프레임 파싱 로직
   - 헤더 핸드셰이크 재검토

3. Authentication Investigator
   - JWT 토큰 만료 시간
   - Refresh token 로직
   - 세션 상태 추적

4. Application Logic Investigator
   - 메시지 핸들러 에러 경로
   - 리소스 누수 가능성
   - 상태 머신 전이

5. Infrastructure Investigator
   - 로드 밸런서 timeout
   - 프록시 설정
   - 방화벽 규칙

Have them actively debate and try to disprove each other's theories.
Share findings in a central document.
```

**작동:**
- 각 팀원이 독립적으로 증거 수집
- 다른 가설을 반박하는 테스트 작성
- 토론을 통한 검증
- 가설 생존자 = 근본 원인

**이득:**
- 단일 조사: 첫 가설에 고착 (앵커링 바이어스)
- 경쟁 가설: 근본 원인 발견율 ↑ 50%

#### 사례 3: 크로스 레이어 리팩토링

**상황:** 인증 모듈 전체 교체 (DB, API, 프론트엔드)

**팀 구성:**
```text
Refactor the authentication system across all layers.
Spawn 3 specialists:

1. Database Specialist
   - 새 스키마 설계
   - 마이그레이션 스크립트
   - 롤백 절차
   - 작업: ["schema_design", "migration_script", "rollback_test", "backup_plan"]

2. API Specialist (이전 작업 의존)
   - 새 엔드포인트 구현
   - 레거시 호환성
   - 토큰 검증 로직
   - 작업: ["new_endpoints", "legacy_compat", "token_validation"]
   - 의존성: [database_specialist]

3. Frontend Specialist (이전 작업 의존)
   - 인증 플로우 업데이트
   - 토큰 저장소 변경
   - 재인증 처리
   - 작업: ["flow_update", "token_storage", "reauth_handling"]
   - 의존성: [api_specialist]

Setup task dependencies so:
- DB work completes first
- API team starts once DB is ready
- Frontend team starts once API is ready
```

**병렬화 이득:**
- **순차 작업:** 3주 (연속적)
- **병렬 작업:** 1.5주 (DB 2일 + API 4일 + FE 3일, 겹침)
- **시간 단축:** 35% (크로스 레이어는 의존성이 많아 완전 병렬화 불가)

### 1.7 성능 및 비용

#### 토큰 비용 분석

**3명 팀 기준 (총 100시간 작업):**

| 지표 | 값 | 설명 |
|------|-----|------|
| **리더 토큰** | 50K | 작업 분석, 조정, 결과 종합 |
| **팀원 1** | 300K | 30시간 작업 × 10K/시간 |
| **팀원 2** | 350K | 35시간 작업 × 10K/시간 |
| **팀원 3** | 400K | 40시간 작업 × 10K/시간 |
| **합계** | **1.1M** | |
| **단일 세션** | **0.8M** | 100시간 순차 작업 |
| **비율** | **1.4배** | 팀 구성 오버헤드 |

**어느 때 가치 있는가:**
- ✅ 시간이 금(Time-to-market)인 경우
- ✅ 복잡한 협력이 필요한 경우
- ✅ 병렬 이득이 토큰 비용 > 경우
- ❌ 순차 작업이 명확한 경우
- ❌ 단순한 반복 작업

#### 실제 성능 수치

**병렬 코드 리뷰 (3명, 100줄 PR):**
```
┌──────────────────────────┐
│ Single Reviewer (30분)   │
├──────────────────────────┤
│ 3-Team Review (15분)     │ ← 50% 시간 단축
│ Lead synthesis (5분)     │
└──────────────────────────┘
```

### 1.8 베스트 프랙티스

#### 1) 팀원에게 충분한 컨텍스트 제공

```text
❌ 나쁜 예:

✅ 좋은 예:
"보안 리뷰어를 생성해줄래.

- 인증 모듈 src/auth/ 리뷰
```

#### 2) 적절한 팀 규모 선택

**팀 규모별 특성:**

| 규모 | 용도 | 토큰 | 조정 오버헤드 |
|------|------|------|-------------|
| **2명** | 간단한 작업 | 2.3배 | 낮음 |
| **3-5명** | 중간 복잡도 (권장) | 3-5배 | 중간 |
| **6-10명** | 복잡한 작업 | 6-10배 | 높음 |
| **10명+** | 매우 복잡 | 10배+ | 매우 높음 |

**경험칙:**
- 팀원당 5-6개 작업 → 최적 효율
- 팀원당 1-2개 작업 → 유휴 팀원 발생
- 팀원당 10개+ 작업 → 병렬화 이득 감소

#### 3) 작업 크기 최적화

**크기 지표:**
```
작업: "인증 모듈 보안 감사"

│  → 분해 필요
│
├─ 분해 1: "JWT 토큰 처리 검증"
│  → 크기: 1-2시간 (적절)
│
├─ 분해 2: "세션 관리 감사"
│  → 크기: 1-2시간 (적절)
│
└─ 분해 3: "입력 검증 확인"
   → 크기: 1시간 (적절)
```

#### 4) 파일 충돌 회피

### 1.9 한계 및 주의사항

#### 기술적 한계

| 한계 | 영향 | 해결 방법 |
|------|------|---------|
| **세션 복구 불가** | `/resume` 후 팀원 손실 | 새로운 팀 재생성 |
| **작업 상태 지연** | 완료 표시 안 됨 | 수동 상태 업데이트 |
| **종료 속도 느림** | 팀원 셧다운 대기 | 인내심 필요 |
| **팀당 1개 리더** | 순차 팀 생성 필수 | 팀 정리 후 재생성 |
| **중첩 불가** | 팀원이 새 팀 생성 불가 | 리더만 생성 |

#### 운영 주의사항

**1) 과도한 토큰 사용**
```text
⚠️ 문제: 10명 팀 × 장시간 작업
→ 토큰 폭증, 비용 급증

✅ 해결:
- 팀 규모 3-5명으로 제한
- 작업 크기 적절히 분할
- 비용 모니터링 (ccusage)
```

**2) 권한 요청 폭증**
```text
⚠️ 문제: 팀원들의 파일 쓰기/명령 실행 시 매번 요청
→ 인터랙션 폭증

✅ 해결:
- Pre-approved operations 설정
- Permission 설정 관대히 구성
```

**3) 파일 충돌**
```text
⚠️ 문제: 같은 파일을 여러 팀원이 동시 수정
→ 덮어쓰기, 손실

✅ 해결:
- 작업 분할 시 파일 경계 명확히
- Lock 기반 협력 (File locking)
```

**4) 좀비 팀원**
```text
⚠️ 문제: 팀원이 에러 후 멈춤
→ 작업 진행 불가

✅ 해결:
- 직접 메시지로 재지시
- 새로운 팀원 스폰 대체
```

#### 효과적이지 않은 경우

```text
❌ Agent Teams 사용하지 말 것:

1. 순차 의존성이 많은 경우
   - A → B → C → D (직선적)
   - 병렬화 불가능

2. 같은 파일을 여러 팀원이 수정해야 하는 경우
   - 충돌 위험 높음

3. 간단한 반복 작업
   - 조정 오버헤드 > 이득

4. 시간이 중요하지 않은 경우
   - 토큰 비용이 더 비쌈

✅ 대신 사용:
- 단일 세션
- Subagents (경량 병렬화)
- Git worktrees (수동 병렬)
```

## 2. Scheduled Tasks / Cron 기능

### 2.1 정의 및 용도

**Scheduled Tasks란:**
- Claude Code 세션 내에서 **프롬프트를 자동으로 반복 실행**하는 기능
- 폴링, 정기 확인, 일회 알림 등에 활용
- **세션 스코프**: 세션 종료 시 작업도 사라짐

**세 가지 스케줄링 방식:**

| 방식 | 실행 위치 | 지속성 | 사용 시기 |
|------|---------|-------|----------|
| **Cloud Tasks** | Anthropic 클라우드 | 영구 | 24/7 자동화 필요 |
| **Desktop Tasks** | 로컬 머신 | 영구 | 로컬 파일 접근 필요 |
| **Session Tasks** | 현재 세션 | 세션 스코프 | 세션 중 임시 폴링 |

**주요 용도:**

1. **배포 상태 폴링**
   ```text
   /loop 5m 배포 상태 확인하고 결과 알려줄래
   ```

2. **PR 감시**
   ```text
   /loop 30m PR #142 승인 상태 체크
   ```

3. **장시간 빌드 모니터링**
   ```text
   /loop 10m 빌드 완료 확인해줄래
   ```

4. **일회 알림**
   ```text
   3시간 후 릴리스 브랜치 푸시 상기시켜줄래
   ```

5. **주기적 작업**
   ```text
   /loop 1d 매일 밤 테스트 스위트 실행
   ```

---

### 2.2 스케줄링 문법 (Cron Expression)

#### 5-Field Cron 표기법

```
┌────────────── minute (0 - 59)
│ ┌──────────── hour (0 - 23)
│ │ ┌────────── day of month (1 - 31)
│ │ │ ┌──────── month (1 - 12 또는 JAN - DEC)
│ │ │ │ ┌────── day of week (0 - 7, 0=7=Sunday 또는 SUN - SAT)
│ │ │ │ │
│ │ │ │ │
* * * * *
```

#### 기본 문법

| 기호 | 의미 | 예시 |
|------|------|------|
| `*` | 모든 값 | `* 9 * * *` = 매일 9:xx |
| `5` | 특정 값 | `5 9 * * *` = 매일 9:05 |
| `*/5` | 간격 | `*/5 * * * *` = 5분마다 |
| `1-5` | 범위 | `0 9 * * 1-5` = 월~금 9:00 |
| `1,15,30` | 목록 | `0 12 1,15,30 * *` = 1,15,30일 정오 |

#### 자주 쓰이는 표현식

```
# 시간 기반
*/5 * * * *      → 5분마다
*/15 * * * *     → 15분마다
0 * * * *        → 매시간 (정각)
0 9 * * *        → 매일 9:00 AM (로컬 시간)
0 12 * * 1-5     → 평일 정오
0 9 * * 0        → 매주 일요일 9:00 AM

# 일 기반
0 0 * * *        → 매일 자정
0 0 1 * *        → 매월 1일
0 0 1 1 *        → 매년 1월 1일

# 고급
0 9,17 * * *     → 매일 9 AM과 5 PM
30 14 15 3 *     → 3월 15일 2:30 PM
0 */6 * * *      → 6시간마다 (0, 6, 12, 18 시)
```

#### `/loop` 자연언어 변환

Claude는 자연언어를 Cron 표현식으로 자동 변환:

```text
입력: /loop 5m 배포 확인
변환: */5 * * * *
설명: 매 5분마다 실행

입력: /loop 30m 체크
변환: */30 * * * *
설명: 30분마다

입력: /loop check every 2 hours
변환: 0 */2 * * *
설명: 2시간마다 (정각)

입력: /loop remind me at 3pm
변환: 0 15 * * *
설명: 매일 3:00 PM

입력: in 45 minutes, check tests
변환: X (45분 후 일회)
설명: 일회만 실행
```

### 2.3 클라우드 vs 데스크탑 vs 세션 스케줄링

#### 비교 표

| 특성 | Cloud | Desktop | Session |
|------|-------|---------|---------|
| **실행 위치** | Anthropic 서버 | 로컬 PC | 현재 세션 |
| **지속성** | ✅ 영구 | ✅ 영구 | ❌ 세션 종료 시 삭제 |
| **머신 전원 필요** | ❌ 아니오 | ✅ 예 | ✅ 예 |
| **최소 간격** | 1시간 | 1분 | 1분 |
| **로컬 파일 접근** | ❌ 불가 | ✅ 가능 | ✅ 가능 |
| **MCP 서버** | 커넥터 설정 | 로컬 설정 | 상속 |
| **권한 요청** | ❌ 없음 | ⚙️ 설정가능 | 상속 |
| **비용** | 토큰 사용만 | 무료 | 무료 |

#### 선택 가이드

```
"정기적으로 git repository를 pull해서 테스트 실행하고 싶어"
→ 로컬 파일 접근 필요 → Desktop 또는 Session

"매일 밤 CI 파이프라인 상태를 확인하고 싶어"
→ 24/7 자동화, 머신 꺼져도 실행 → Cloud

"지금 세션에서 배포 상태 계속 폴링하고 싶어"
→ 임시, 세션 중만 필요 → Session (/loop)
```

### 2.4 트리거 타입 (시간, 이벤트, 조건부)

### 2.5 실행 환경 및 권한

#### 세션 작업의 실행 환경

```
┌─────────────────────────────┐
│   Claude Code Session       │
├─────────────────────────────┤
│ ┌───────────────────────┐   │
│ │  Main Turn (Active)   │   │
│ │  (사용자와 대화)      │   │
│ └───────────────────────┘   │
│         ▲                    │
│         │                    │
│         └──────┬─────────────┤
│                │             │
│  ┌─────────────▼──────────┐ │
│  │ Scheduled Prompt       │ │
│  │ (백그라운드 대기)      │ │
│  │ - 낮은 우선순위        │ │
│  │ - 메인 턴 끝날 때 실행 │ │
│  └────────────────────────┘ │
│                              │
│  ┌─────────────────────────┐│
│  │ Another Scheduled Prompt││
│  └─────────────────────────┘│
└──────────────────────────────┘
```

**실행 타이밍:**
- Claude가 장시간 응답 중 → 예약 작업 대기
- Claude 응답 완료 → 예약 작업 실행
- 응답 중 작업 기한 지남 → 다음 공백 시 실행 (Catch-up 없음)

#### 권한 설정

**세션 작업은 세션의 권한을 상속:**

```json
{
  "permission_mode": "ask",        // 각 작업마다 권한 요청
  "permission_mode": "auto",       // 자동 허용 (위험)
  "permission_mode": "skip"        // 권한 요청 없음
}
```

**Desktop/Cloud 작업: 작업별 권한 설정 가능**

```yaml
tasks:
  - name: run-tests
    schedule: "0 2 * * *"
    permission_mode: auto          # 테스트 실행은 자동
  - name: deploy-prod
    schedule: "0 9 * * 1"
    permission_mode: ask           # 배포는 물어보기
```

#### 타임존 해석

```
/loop 0 9 * * * 매일 9:00 AM 테스트 실행

→ 로컬 타임존 기준으로 해석됨
   서울 (UTC+9): 9:00 KST
   샌프란시스코 (UTC-8): 9:00 PST
   런던 (UTC+0): 9:00 GMT
```

### 2.7 에러 처리 및 재시도

#### Claude Code의 기본 동작

```
작업 실행 실패
    ↓
에러 로그에 기록
    ↓
다음 일정 시각에 다시 시도
    ↓
Catch-up 없음 (기한 지난 실행은 스킵)
    ↓
재시도 정책 없음 (기본 동작 아님)
```

#### 에러 처리 전략

**전략 1: 자체 에러 처리**

```text
/loop 5m 배포 상태를 확인하고, 
실패하면 이전 상태와 비교해서 
상황이 악화되지 않았으면 조용히 넘어가, 
악화되면 알림을 보내줄래.
```

**전략 2: 재시도 래퍼**

```bash
#!/bin/bash
# retry-wrapper.sh
max_retries=3
retry_count=0

while [ $retry_count -lt $max_retries ]; do
  if ./deploy-check.sh; then
    exit 0
  fi
  retry_count=$((retry_count + 1))
  sleep 30
done

exit 1  # 최종 실패
```

```text
/loop 30m ./retry-wrapper.sh
```

**전략 3: 상태 추적으로 연속 실패 감지**

```javascript
// state.json
{
  "last_check": "2026-03-31T14:00:00Z",
  "last_status": "pending",
  "failure_count": 3,
  "last_success": "2026-03-31T13:00:00Z"
}
```

```text
/loop 5m 배포 상태를 확인하고, 
state.json을 업데이트해줄래.
연속 실패가 5회 이상이면 경고를 보내줄래.
```

#### 제한사항

**Claude Code는 자동 재시도 미지원:**

```

✅ 해결:

또는 Desktop/Cloud 작업 사용 (재시도 지원)
```

### 2.9 성능 및 제한사항

#### 세션 스코프 작업의 제한

| 제약 | 영향 | 해결 |

| **3일 자동 만료** | 반복 작업 자동 삭제 | 재생성 필요 |

#### 성능 특성

**지연시간:**
```
예약 시각 → 작업 실행: 최대 1분
```

**Jitter (편차):**
```
반복 작업:
- 주기의 최대 10% 지연
- 최대 15분
- 예: 1시간 작업 → ±6분 지연 가능

일회 작업:
- 정각/반시간의 ±90초 지연
- 예: 3:00 PM → 14:58:30 ~ 15:01:30
- 영향: 정각이 아닌 분 선택 시 Jitter 제거
  (예: 3:07 PM → Jitter 없음)
```

**토큰 비용:**
```
작업당 평균: 1-5K 토큰
- 설정: 0.5K
- 실행 후처리: 0.5K
- 프롬프트 실행: 0.5-4.5K

배포 상태 확인 (2분 간격, 30분 진행):
├─ 총 15회 실행
├─ 평균 2K/실행 = 30K 토큰
└─ 비용: 약 $0.005 (대략)
```

#### 1) 적절한 간격 선택

```text
❌ 너무 빈번:
/loop 10s 상태 확인 (너무 자주 API 호출)

⚠️ 중간:

✅ 좋음:
/loop 5m 상태 확인 (효율적)

- 일일 작업: 1-24시간
```

```text
❌ Jitter 영향받는 경우:

45 14 * * *  → 2:45 PM은 지연 없음

규칙: 정각(00)이나 반시(30)이 아닌 분 선택
```

#### 3) 작업 명명 및 추적

```text
좋은 명명:
- "배포-상태-폴링"
- "PR-142-리뷰-감시"
- "일일-테스트-스위트"
- "야간-성능-리포트"

추적:
작업 리스트 출력
> 5개 작업 실행 중
- 배포-상태-폴링 (2분마다)
- PR-142-리뷰-감시 (15분마다)
- 빌드-모니터링 (5분마다) ← 최근 실패
- 일일-테스트-스위트 (23:59 내일)
- 야간-성능-리포트 (01:00 내일)
```

#### 4) 에러 기록 및 모니터링

```text
/loop 30m 배포 상태를 확인하고,
실패 여부를 deployment-checks.log에 기록해줄래.
연속 3회 실패 시 경고를 보내줄래.
```

**구현:**
```bash
#!/bin/bash
# deployment-check.sh

status=$(check_deployment)
timestamp=$(date -u +"%Y-%m-%dT%H:%M:%SZ")

if [ "$status" != "complete" ]; then
  echo "[$timestamp] FAIL: $status" >> deployment-checks.log
  
  failures=$(grep -c "FAIL" deployment-checks.log | tail -3)
  if [ "$failures" -ge 3 ]; then
    send_alert "배포 체크 3회 연속 실패"
  fi
else
  echo "[$timestamp] SUCCESS: $status" >> deployment-checks.log
fi
```

#### 5) 비용 최적화

```text
고비용 시나리오:
├─ 60개 작업 × 2K 토큰 = 120K/실행
├─ 매시간 = 2.88M/일

1. 중요한 작업만: 가장 효과적
2. 간격 늘리기: 5분 → 10분 (50% 절감)
3. 조건부 실행: 필요시만 (편차 큼)

```

## 3. 비교 분석 및 선택 가이드

### 3.1 Agent Teams vs Subagents

**언제 어떤 걸 쓸까?**

```
작업: "큰 코드베이스 병렬 리팩토링"

Agent Teams:
├─ 팀원끼리 대화 O
├─ 작업 공유 O
├─ 비용 높음 (3-5배)
├─ 조정 복잡함
└─ 결과: 매우 좋음 (협력)

Subagents:
├─ 팀원끼리 대화 X
├─ 각각 독립 실행
├─ 비용 낮음 (2-3배)
├─ 간단함
└─ 결과: 좋음 (병렬화)
```

**선택 매트릭스:**

| 작업 특성 | Agent Teams | Subagents |
|----------|------------|----------|
| **협력 필요도 높음** | ✅ | ❌ |
| **독립적 작업** | ❌ | ✅ |
| **토큰 비용 중요** | ❌ | ✅ |
| **시간 중요** | ✅ | ⚠️ |
| **복잡한 조정** | ✅ | ❌ |

### 3.2 Scheduled Tasks의 3가지 모드

```
선택 결정트리:

Q1: 24/7 자동화 필요?
├─ YES → Cloud 사용 (머신 꺼도 실행)
└─ NO → Q2로

Q2: 로컬 파일 접근 필요?
├─ YES → Desktop (git, npm 등)
└─ NO → Q3로 또는 Cloud

Q3: 세션 중 임시로만 필요?
├─ YES → Session (/loop) (간단, 빠름)
└─ NO → Desktop (영구 저장)
```

## 요약 및 권장사항

### 🎯 핵심 정리

**Agent Teams:**
- **언제**: 팀원 간 협력이 중요한 복잡한 작업
- **장점**: 높은 품질, 빠른 속도, 자동 조정
- **비용**: 높음 (3-5배)
- **학습곡선**: 높음 (초기 설정 복잡)
- **ROI**: 시간 가치가 높으면 긍정적

**Scheduled Tasks:**
- **언제**: 반복적인 모니터링 및 자동화
- **장점**: 간단함, 유연함, 저비용
- **비용**: 낮음 (실행 횟수 기반)
- **학습곡선**: 낮음 (직관적)
- **ROI**: 항상 긍정적 (시간 절감)
